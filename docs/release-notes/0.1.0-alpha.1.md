# aiignore policy 0.1.0-alpha.1

This is the first experimental public draft and reference implementation. It is
published for interoperability testing and security review, not as an endorsed
industry standard.

## Included

- Normative `.aiignore.yaml` draft 0.1 specification and JSON Schema.
- TypeScript parser, deterministic decision engine, and CLI.
- Scoped `@apinindy/aiignore` npm package with the `aiignore` executable, public
  TypeScript API, schemas, profiles, and portable conformance artifacts.
- 107 language-neutral decision vectors and 56 parser vectors, enforced by a
  schema/runtime coverage contract.
- Content-addressed decision, parser, and live-harness vector documents that
  self-identify their canonical URI and are reported with exact SHA-256 values.
- A versioned conformance manifest binding the complete draft 0.1 artifact set
  by role, canonical URI, package release, and SHA-256.
- Closed protocol-token registries and an immutable errata index included in
  the canonical conformance bundle.
- Safe `aiignore init` onboarding that installs the reviewed recommended
  profile without overwriting an existing or legacy compatibility file.
- Portable generated-candidate vectors and deterministic candidate, resource
  matcher, string matcher, transformed-output, and replacement-count limits.
- Fail-closed hook traversal budgets, package-pinned hook execution, validated
  vendor responses, and secret-safe structured audit events.
- Hook path resolution that preserves the harness event working directory while
  evaluating the resulting absolute path against the pinned policy root.
- Bounded plural path/URL array mediation for bulk tool inputs, including
  Gemini CLI `read_many_files`, plus conservative mutation/discovery aliases.
- Explicit partial-compilation reporting when Codex's safer local/private
  network defaults narrow portable network default-allow semantics.
- Race-resistant, no-follow, size-bounded policy and conformance-artifact reads,
  including growth during the read, plus strict empty-digest pin rejection.
- Detached Ed25519 conformance-report signatures with exact-byte verification
  and mandatory out-of-band identity and public-key trust pins.
- Installed-package production license verification that cross-checks the
  authenticated lock graph rather than trusting editable license labels.
- Reproducible parser/decision robustness fuzzing on every verification run and
  a scheduled rotating-seed extended campaign.
- Checksum-pinned Gitleaks scanning across every reachable commit on pull
  requests, schedules, and releases, with repository suppression mechanisms
  disabled for the required scan.
- Two-phase publication that binds the protected workflow dispatch to the exact
  signed tag, proves it is on `main`, verifies every canonical Pages artifact
  byte-for-byte, attests the checksum file, and only
  promotes the GitHub prerelease after matching npm `alpha` publication;
  phase two verifies rather than replaces the inspected draft, the SBOM is
  normalized for reproducible bytes, title and notes are authenticated, and
  immutable historical-release reruns require a matching npm SHA-512 integrity
  value without moving dist-tags or comparing the historical tag against a
  later, mutable Pages deployment.
- Release privilege separation: dependency installation, verification, package
  construction, and SBOM generation run without OIDC or write authority; the
  protected release job accepts only a re-hashed inert inventory and a pinned
  npm publication CLI, and cannot execute build commands.
- Versioned GitHub hosting desired state and a read-only, secret-name-safe audit
  for branch/tag rulesets, public security products, Pages, immutable releases,
  workflow permissions, and the tag-restricted release environment.
- Release authority is explicitly single-maintainer for this alpha: no
  independent human approval is claimed. Administrator bypass remains disabled,
  `v*` creation is restricted to the documented owner, and publication is bound
  to a verified tag with the exact maintainer identity and release message.
- A versioned traceability catalog maps all 16 normative sections and 151 BCP 14
  keyword occurrences to checked-in evidence and non-removable external limits.
- Executable boundary witnesses cover all 86 primitive policy-schema assertion sites;
  catalog evidence paths resolve against the immutable release-tag source tree.
- Rotating-seed extended fuzzing on every pull request as well as the weekly
  schedule, keeping the required check present for documentation-only changes.
- Verbatim DCO 1.1 and a dedicated pull-request gate that validates every
  commit sign-off against its author name and email without executing the
  contributor-controlled validator or checkout.
- Strict release-tag input grammar and machine-enforced ownership, intake,
  compatibility, conformance-evidence, credential, support, and OSPS
  self-assessment contracts.
- Monotonic npm `alpha` publication that rejects an older unpublished signed
  tag before it can roll the global prerelease channel backward, while
  immutable historical reruns leave dist-tags unchanged.
- A security-architecture inventory and control-by-control OSPS Baseline
  2026.02.19 self-assessment covering all 64 controls without implying a badge
  or deployed conformance.
- Versioned JSON Schemas for exact portable decision objects and minimal
  secret-safe audit events, with the conformance runner validating every result
  and adapters reporting canonicalization failures separately instead of
  fabricating deny-rule matches. A denied multi-resource operation does not
  emit a misleading audit event for a crossing that never occurred.
- A deterministic `aiignore doctor` command and readiness-report schema that
  expose permissive defaults, missing control families, and adapter gap counts
  without logging policy paths, patterns, rule IDs, or candidate material—and
  structurally refuse to claim that deployment enforcement is established.
- A deterministic reference-conformance command and exact implementation-report schema
  that execute all 163 manifest-selected parser/decision cases, distinguish
  reference/derived/independent claims, and cannot be confused with a live
  harness-enforcement report. Detached signatures use a separate payload type
  and domain separator for the new report class; an offline verifier rejects
  manifest substitution, suite omission, and vector-identity drift.
- Experimental Codex and Gemini CLI adapters with explicit gap reports.
- A content-addressed provisional Codex 0.144.5/macOS Seatbelt report with ten
  passing controls and the readable pre-existing archive retained as a failure.
- Threat model, adoption guidance, governance process, and recommended profile.
- OpenSSF Security Insights 2.2.0 metadata, a transparent OSPS baseline
  self-assessment, and machine-enforced production dependency licenses.

## Important limitations

- Exact `.aiignore` remains a gitignore-style compatibility filename and is not
  the structured policy format.
- No independent implementation or independent security audit exists yet.
- No separate standards-essential patent or certification-mark policy exists;
  the repository is currently MIT licensed with DCO sign-off.
- Adapter reports are scoped evidence, not vendor endorsement or a universal
  security guarantee.
- Filesystem and network isolation require a tested harness sandbox or external
  enforcement layer; the reference CLI alone does not install one.
- Archive, cache, VCS-object, redirect, DNS, browser, MCP, and alternate-tool
  paths require harness-specific mediation and testing.

## What feedback is most useful

- Independent implementations that run the published parser and decision
  vectors and report ambiguous clauses.
- Versioned harness tests that identify which context, tool, subprocess,
  filesystem, and network paths are actually mediated.
- Adversarial review of YAML safety, glob and URL portability, Unicode and
  platform aliases, policy integrity, string-filter limits, and adapter bypasses.
- Concrete filename, migration, and interoperability experience from existing
  AI ignore-file implementations.

Use the RFC template for normative proposals and private vulnerability
reporting for security issues. Do not include real credentials or private
harness transcripts in public reports.

Verify release assets against the attached SHA-256 file and GitHub artifact
attestation. See `SECURITY.md` before deployment.
